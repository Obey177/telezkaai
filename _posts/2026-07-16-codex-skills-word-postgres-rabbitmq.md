Слово **skill** сначала кажется ещё одним лишним англицизмом. Но смысл у него простой: один раз объясняешь Codex, как выполнять работу, а потом не повторяешь всю инструкцию заново.

В Codex это действительно полезная штука. Skill — это сохранённый способ выполнить конкретную работу. Внутри могут лежать правила, шаблоны, справочные материалы и, когда без них никак, небольшие программы для обращения к другим системам.

> Хорошая автоматизация начинается не с робота. Она начинается с момента, когда человеку надоедает в двадцатый раз объяснять одно и то же.

В этой статье разберём три самостоятельных примера:

1. Codex получает Excel-файл и готовит Word-отчёт по корпоративному шаблону.
2. Codex обращается к PostgreSQL и объясняет состояние записи человеческим языком.
3. Codex поднимает локальный RabbitMQ, находит сообщение по ID и выводит его `description`.

Каждый пример лежит в отдельном архиве. Его можно скачать, распаковать, открыть как локальную папку в Codex и повторить сценарий без сборки общего демонстрационного проекта.

---

## Что такое skill в этом примере

Codex ищет навыки проекта в папке `.agents/skills/`. У каждого навыка есть обязательный файл `SKILL.md`: в нём написано, когда применять навык, какие материалы использовать, какие действия выполнить и каким должен быть результат.

Простейший skill может состоять только из текста и шаблона. Более технический skill может дополнительно содержать C#-коннектор, SQL-схему или утилиту для RabbitMQ.

В наших архивах встречаются три варианта:

Официально skills доступны в приложении ChatGPT, Codex CLI и расширении для IDE. В этой статье используем самый наглядный вариант для обычного пользователя: **приложение ChatGPT с выбранным режимом Codex**. Подробнее устройство навыков описано в [официальной документации Codex](https://developers.openai.com/codex/build-skills).

### Общая схема запуска

Для всех трёх примеров начало одинаковое:

1. Скачайте нужный архив по ссылке в статье.
2. Распакуйте его в отдельную папку.
3. Запустите приложение ChatGPT и войдите в аккаунт.
4. Создайте новую задачу и выберите **Codex**.
5. Откройте распакованную папку как локальный проект.
6. Отправьте запрос из соответствующего раздела ниже.
7. Разрешите запуск локальных команд, если Codex запросит подтверждение.

Codex увидит папку `.agents/skills`, сопоставит запрос с описанием навыка и прочитает полный `SKILL.md`. Навык можно вызвать явно: напишите его имя со знаком `$`, например `$weekly-operations-report`. В приложении и CLI можно также ввести `$` и выбрать нужный skill из списка.

На Linux тот же проект можно открыть через Codex CLI: перейти в папку проекта, выполнить `codex` и отправить тот же запрос.

Здесь нет приложения, которое надо собирать или запускать. Сам skill хранит:

- порядок обработки таблицы;
- формулы для основных показателей;
- требования к формулировкам;
- Word-шаблон;
- список полей, которые нужно заполнить;
- пример готового документа.

### Порядок действий пользователя

#### Шаг 1. Распакуйте архив

Распакуйте `codex-skill-01-word-report.zip` в любую удобную папку. Не переносите отдельно только Excel или Word: Codex должен видеть всю структуру, включая скрытую папку `.agents`.

#### Шаг 2. Запустите Codex

Откройте приложение ChatGPT, выберите **Codex** и укажите распакованную папку `codex-skill-01-word-report` как рабочую.

#### Шаг 3. Отправьте запрос

```text
$weekly-operations-report

Подготовь недельный операционный отчёт по файлу data/sales_by_day.xlsx.
Используй корпоративный Word-шаблон из skill и сохрани результат
в output/weekly_operations_report.docx.
```

Такой запрос специально написан обычным языком. Пользователю не нужно объяснять формулы, называть поля Word-шаблона или перечислять все проверки: это уже записано в skill.

#### Шаг 4. Проверьте план действий

Codex должен:

1. прочитать таблицу за четырнадцать дней;
2. сравнить последние семь дней с предыдущими семью;
3. посчитать заказы, выручку, успешность, средний чек и возвраты;
4. найти заметные дневные отклонения;
5. создать копию Word-шаблона;
6. заполнить таблицы и текстовые блоки;
7. сохранить новый документ в папку `output`.

Если Codex предлагает изменить исходный шаблон вместо создания копии, попросите его следовать `SKILL.md`: оригинальный файл должен остаться нетронутым.

#### Шаг 5. Откройте результат

Готовый документ появится здесь:

```text
output/weekly_operations_report.docx
```

В демонстрационных данных текущая неделя содержит:

- 2 426 заказов;
- 9 147 680 ₽ выручки;
- 95,9% успешных операций;
- средний чек 3 933 ₽;
- 141 095 ₽ возвратов.

Главное отклонение — 10 июля: 28 ошибок, или 7% дневных заказов. Skill отдельно требует не придумывать причину. Поэтому хороший вывод будет звучать так: **нужно проверить логи платёжного шлюза и antifraud-системы**, а не «система точно сломалась из-за antifraud».

> Умный помощник отличается от пьяницы в подворотне тем, что не выдаёт догадки за факты. Не знает причину — так и говорит, а не сочиняет её по дороге.

### Что здесь показывает skill

Этот пример важен именно отсутствием отдельного приложения. Skill не обязан быть программой. Иногда достаточно один раз сохранить рабочую инструкцию, шаблон и критерии качества — и больше не объяснять каждую неделю, где должна стоять выручка и почему руководителю не нужен роман на двенадцать страниц.

- включает режим транзакций только для чтения.

#### Шаг 1. Распакуйте архив и запустите Docker Desktop

Распакуйте архив в отдельную папку. Затем запустите Docker Desktop и убедитесь, что Docker Engine работает.

Самостоятельно вводить команды необязательно: дальше их может выполнить Codex.

#### Шаг 2. Откройте папку в Codex

В приложении ChatGPT выберите **Codex** и откройте папку `codex-skill-02-postgres-status`.

#### Шаг 3. Отправьте запрос

```text
$db-record-status

Подними локальный PostgreSQL из docker-compose.yml и проверь запись ORD-1042.
Покажи текущий статус, описание, ответственного, время обновления
и короткую историю изменений.
Ничего в базе не изменяй.
```

Последняя фраза не обязательна — запрет на запись уже есть в skill и в самой базе. Но для демонстрации она полезна: человек сразу видит границу полномочий агента.

#### Шаг 4. Подтвердите запуск локальных команд

Codex прочитает `SKILL.md` и предложит выполнить команды примерно такого вида:

```bash
docker compose up -d --wait postgres
```

```bash
dotnet run \
  --project .agents/skills/db-record-status/scripts/DbRecordStatus.csproj \
  -- --id ORD-1042
```

Пользователю не требуется запоминать эти команды. Они уже записаны в skill. Нужно только проверить, что Codex работает в правильной папке, и разрешить безопасный локальный запуск.

#### Шаг 5. Получите ответ

Ожидаемый результат выглядит так:

```text
Запись:        ORD-1042
Тип:           order
Статус:        PAYMENT_REVIEW
Описание:      Платёж получен, заказ ожидает ручной antifraud-проверки.
Ответственный: risk-team
Обновлена:     13.07.2026 16:42:00 +03:00
```

История показывает три последовательных состояния:

```text
CREATED
PAYMENT_RECEIVED
PAYMENT_REVIEW
```

Codex может пересказать это обычным языком:

> Заказ создан, подтверждение оплаты получено, но сейчас он остановлен на ручной antifraud-проверке. За проверку отвечает команда `risk-team`. Последнее изменение произошло 13 июля 2026 года в 16:42.

#### Шаг 6. Остановите стенд

После демонстрации отправьте Codex ещё один запрос:

```text
Останови локальный PostgreSQL этого проекта, но сохрани данные контейнера.
```

За кулисами будет выполнено:

```bash
docker compose down
```

Для полного удаления демонстрационной базы и volume используется `docker compose down -v`.

### Что здесь показывает skill

Skill не хранит пароль от боевой базы и не учит модель придумывать SQL на лету. Он задаёт проверенный маршрут:

1. прочитать известную схему;
2. запустить готовый коннектор;
3. использовать read-only пользователя;
4. вернуть только нужные поля;
5. не печатать чувствительные метаданные без необходимости.

Такой подход полезен для заказов, заявок, счетов, возвратов и любых внутренних сущностей, состояние которых сотрудники регулярно ищут вручную.

## Пример 3. Найти сообщение в RabbitMQ по ID

Очередь сообщений — место, где всё выглядит спокойно ровно до той минуты, когда кто-то спрашивает: «а сообщение `MSG-7F31A9` вообще было?»

В архиве находится локальный RabbitMQ, несколько демонстрационных сообщений и C#-утилита. Она умеет загрузить сообщения в очередь, найти нужное по ID и вывести только полезное поле `description`. В live-режиме сообщения не удаляются: утилита просит RabbitMQ вернуть просмотренную выборку в очередь и дополнительно показывает количество сообщений до и после проверки.

### Порядок действий пользователя

#### Шаг 1. Распакуйте архив и запустите Docker Desktop

Распакуйте `codex-skill-03-rabbitmq-message.zip`, затем запустите Docker Desktop.

```text
$rabbit-message-by-id

Подними локальный RabbitMQ из docker-compose.yml,
```

#### Шаг 4. Разрешите подготовку стенда

Codex выполнит две группы действий. Сначала поднимет RabbitMQ:

```bash
docker compose up -d --wait rabbitmq
```

Затем создаст exchange и очереди и загрузит тестовые сообщения:

```bash
dotnet run \
  --project .agents/skills/rabbit-message-by-id/scripts/RabbitMessageTools.csproj \
  -- seed --file data/rabbit_messages.jsonl --exchange codex.demo --purge
```

#### Шаг 5. Подтвердите ограниченный просмотр без удаления сообщений

Skill требует отдельного подтверждения перед чтением живой очереди. Диагностическая команда использует режим `ack_requeue_true`: RabbitMQ временно отдаёт ограниченную выборку для просмотра, а затем возвращает сообщения в очередь. Они не подтверждаются как обработанные и не удаляются. Именно так `ack_requeue_true` описан в [официальной документации RabbitMQ](https://www.rabbitmq.com/docs/http-api-reference#post-queues-vhost-name-get).

Когда Codex попросит разрешение, ответьте:

```text
Да, разрешаю ограниченный live-просмотр локальной очереди order.events.
Просмотри не больше 20 сообщений. Ничего не удаляй:
после проверки сообщения должны остаться в очереди.
```

После этого Codex запустит поиск:

```bash
dotnet run \
  --project .agents/skills/rabbit-message-by-id/scripts/RabbitMessageTools.csproj \
  -- find --source live --yes-live \
  --queue order.events --id MSG-7F31A9 --count 20
```

#### Шаг 6. Получите результат

```text
Очередь:      order.events
Message ID:   MSG-7F31A9
Routing key:  payment.review.requested
Description:  Повторная проверка статуса оплаты заказа ORD-1042:
              сработало antifraud-правило R-17.
Просмотрено:  3
Режим:        сообщения возвращены в очередь (ack_requeue_true)
В очереди:    до 3, после 3
```

В локальном стенде до поиска в `order.events` было три сообщения и после осталось три. Утилита не только включает возврат сообщений, но и проверяет размер очереди до и после диагностического чтения.

Пользователю нужен был один ответ, поэтому skill не вываливает в чат весь payload, headers и технический мусор. Он возвращает очередь, ID, routing key, `description`, число просмотренных сообщений и подтверждение, что выборка осталась в очереди.

### Безопасный вариант без RabbitMQ

Для первой демонстрации можно вообще не поднимать брокер. В архиве есть локальная выгрузка `data/rabbit_messages.jsonl`.

Запрос будет таким:

```text
$rabbit-message-by-id

Не запускай RabbitMQ. Найди сообщение MSG-7F31A9 в локальной выгрузке
очереди order.events и выведи description.
```

В этом режиме Codex использует команду `find --source mock`. Она ничего не читает из брокера и подходит для безопасного показа на статье, встрече или презентации.

### Важное ограничение

RabbitMQ — не поисковая база данных. Если команда просмотрела первые двадцать сообщений и не нашла нужное, это ещё не означает, что сообщения нет во всей очереди. Поэтому skill обязан сообщать размер просмотренной выборки и не делать слишком уверенных выводов.

В этом примере просмотренные сообщения возвращаются в очередь и не удаляются. Но такой requeue — не настоящий режим «подсмотреть, ничего не трогая»: если с очередью одновременно работают другие потребители, положение сообщений может измениться. На нашем локальном стенде других потребителей нет, поэтому после проверки ожидаем то же количество сообщений.

Для постоянного производственного поиска лучше сохранять `message_id`, статус и нужные поля в отдельном индексируемом хранилище. Live-режим этого примера предназначен для локальной диагностики, а не для регулярного обхода боевой очереди.

> В хорошей системе сообщение проходит очередь и исчезает из твоей жизни. В плохой — остаётся только его ID, ночной дежурный и слабый запах перегретого пластика.

## Что в итоге получает обычный пользователь

Во всех трёх случаях человек формулирует задачу человеческим языком:

- «собери отчёт по шаблону»;
- «скажи, что сейчас с заказом»;
- «найди сообщение и покажи описание».

Техническая последовательность остаётся внутри skill. Пользователю не нужно помнить расположение шаблона, SQL-схему, строку подключения, путь к C#-проекту или параметры диагностической команды.

При этом skill не превращает Codex в бесконтрольного шамана. Наоборот, он задаёт границы:

- исходный Word-шаблон не изменять;
- причины отклонений не придумывать;
- базу открывать только для чтения;
- запись искать по точному ID;
- live-просмотр очереди выполнять только после подтверждения;
- не показывать лишние данные.

Именно в этом практический смысл skills: **не просто научить агента что-то делать, а заранее определить, как именно он должен это делать и где обязан остановиться**.
